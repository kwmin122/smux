## Security Hardening

### Task 31: Socket Permissions + Peer Credential Validation

> Remediates: C-3 (unauthenticated IPC socket)

**What:**
- Set Unix socket file permissions to `0o600` immediately after bind
- Implement `SO_PEERCRED` / `getpeereid()` peer validation on accepted connections
- Reject connections from UIDs that do not match the daemon's UID
- macOS: use `getpeereid()` (SO_PEERCRED is Linux-only); abstract behind a platform trait
- Log rejected connection attempts with peer UID

**Tests:**
- Socket file created with `0o600` permissions (stat check)
- Connection from same UID succeeds
- Unit test for peer credential extraction function
- Integration test: daemon rejects forged UID (if testable; otherwise manual verification documented)

### Task 32: Restrictive CSP in tauri.conf.json

> Remediates: C-2 (CSP disabled)

**What:**
- Set Content-Security-Policy in `crates/smux-app/src-tauri/tauri.conf.json`
- `default-src 'self'`
- `script-src 'self'` (no `unsafe-inline`, no `unsafe-eval`)
- `style-src 'self' 'unsafe-inline'` (required for xterm.js inline styles)
- `connect-src 'self' ipc: tauri:`
- `img-src 'self' data:`
- `font-src 'self' data:`
- Verify Vite build still works with CSP (no inline scripts in production build)
- If Vite injects inline scripts, configure Vite to use external script files or nonces

**Tests:**
- `tauri.conf.json` contains CSP string
- App loads without CSP violation errors in WebView console
- xterm.js renders correctly with CSP active
- Inline script injection via terminal output is blocked

### Task 33: SafetyConfig Enforcement + Server-Side max_rounds Cap

> Remediates: H-3 (SafetyConfig not applied), M-4 (no server-side max_rounds cap)

**What:**
- Daemon's `spawn_session()` uses `create_adapter_with_safety()` instead of `create_adapter()` for all adapter creation
- Load `SafetyConfig` from `~/.smux/config.toml` safety section
- Add server-side `max_rounds` cap in daemon (default: 50), enforced regardless of client-requested value
- If client requests `max_rounds > cap`, clamp to cap and log warning
- Document the cap in config file comments

**Tests:**
- Daemon creates adapters via `create_adapter_with_safety()`
- SafetyConfig values propagated to adapter (unit test with mock)
- `max_rounds` clamped when client exceeds cap
- Session terminates at max_rounds even if planner keeps iterating

### Task 35: Full UUID Session IDs + IPC Message Size Reduction

> Remediates: H-1 (8-char session ID brute-forceable), M-2 (16MiB message limit)

**What:**
- Replace 8-character random session IDs with full UUID v4 (128-bit)
- Use `uuid` crate (`Uuid::new_v4().to_string()`)
- Update all session ID references: daemon, IPC, CLI, Tauri commands
- Reduce IPC message size limit from 16MiB to 1MiB
- Add graceful error when message exceeds limit (instead of silent truncation or panic)
- Update existing tests that hardcode 8-char session IDs

**Tests:**
- Generated session IDs are valid UUID v4 format
- IPC rejects messages larger than 1MiB with descriptive error
- Existing session management works with longer IDs
- CLI displays full UUID (or truncated display with full ID in verbose mode)

## Spec Alignment

### Task 36: Parallel Verifier Execution with tokio::JoinSet

> Remediates: VG-001 (sequential verifier execution)
> **Blocks:** Task 40

**What:**
- Replace sequential verifier loop in `Orchestrator::run_round()` with `tokio::JoinSet`
- Each verifier runs as a spawned task; results collected when all complete
- Timeout per verifier (configurable, default 5 minutes) â€” if one verifier hangs, others still complete
- Timed-out verifiers produce a synthetic `NeedsInfo` result with reason "verifier timed out"
- Emit per-verifier `AgentOutput` events as they stream (not batched at end)
- Single-verifier mode: JoinSet with one task, identical behavior to current

**Tests:**
- 2 verifiers run concurrently (wall-clock time < sum of individual times)
- Timed-out verifier produces NeedsInfo result
- Single-verifier regression: output identical to v0.3 behavior
- All verifier outputs streamed to UI in real-time

### Task 39: Combined Verifier Feedback for Planner

> Remediates: VG-007 (planner only sees first verifier's feedback)

**What:**
- When consensus is `Rejected`, collect rejection reasons from **all** rejecting verifiers
- Build combined feedback string: each verifier's name + rejection category + reason
- Pass combined feedback to planner for next iteration (not just first verifier)
- Format: `"[claude] type_error: missing return type\n[codex] logic_error: off-by-one in loop"`
- For `NeedsInfo` results, aggregate all questions

**Tests:**
- 2/3 verifiers reject: planner receives both rejection reasons
- Combined feedback preserves verifier attribution (name prefix)
- Single-verifier rejection: feedback unchanged from current behavior
- Mixed reject + NeedsInfo: both included in combined feedback

### Task 42: Gemini Adapter Availability Detection + Binary Fallback

> Remediates: deferred from Task 25

**What:**
- On adapter creation, probe `npx @google/gemini-cli --version` with 10-second timeout
- If npx not available, try `gemini --version` (globally installed binary)
- If neither available, return `Err` from `create_adapter()` with clear message
- Daemon: if Gemini requested but unavailable, exclude from verifier list and log warning
- If all requested verifiers unavailable, fail session with error (do not start with zero verifiers)
- Cache availability result for session lifetime (do not re-probe each round)

**Tests:**
- Adapter creation succeeds when `gemini-cli` is available (mock)
- Adapter creation falls back to `gemini` binary when npx fails
- Unavailable adapter excluded from verifier list with warning
- Session fails if all verifiers unavailable
