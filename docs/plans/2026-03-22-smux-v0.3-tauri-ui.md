# smux v0.3 Tauri UI Implementation Plan

> **Goal:** Wrap smux-core in a native macOS app with Tauri v2. Split-pane terminal UI, Focus/Control/Rewind/Compare modes, embedded browser, panel drag-and-drop.

**Architecture:** Tauri v2 app (`crates/smux-app/`) with React + TypeScript frontend. Connects to smux-daemon via Unix socket IPC. Terminal panels use xterm.js. Browser panel uses Tauri's WebView.

**Design:** `docs/design/DESIGN.md` — 3 themes (Deep Navy default, Amber, Forest Green), 9 Stitch reference screens in `docs/design/screens/`

**Tech:** Tauri v2, React 19, TypeScript, Vite, xterm.js, Tailwind CSS v4, Material Symbols, Inter + JetBrains Mono

### Task 25: Gemini CLI Adapter (independent, parallelizable)

### Task 26: Orchestrator Refactor + Types Migration (independent, parallelizable)

**What:**
- Refactor `Orchestrator` struct: `verifier: Box<dyn AgentAdapter>` → `verifiers: Vec<Box<dyn AgentAdapter>>`
- Update `Orchestrator::new()` constructor signature
- Migrate `SessionMeta.verifier: String` → `SessionMeta.verifiers: Vec<String>`
- Add `consensus_strategy` field to `SessionMeta`
- Add `cross_verify: Option<ConsensusResult>` to `RoundSnapshot`
- Run all verifiers in parallel (`tokio::join!`)
- Update daemon's `spawn_session()` to create Vec of adapters
- Backward compatible: single verifier behaves identically to current

**Tests:**
- 1-verifier mode unchanged (regression test)
- 2-verifier parallel execution
- 3-verifier parallel execution with mixed verdicts

---

### Task 27: Consensus Engine (independent, parallelizable)

**What:**
- `ConsensusEngine` in `consensus.rs` (new module)
- 4 strategies: Majority (default), Weighted, Unanimous, LeaderDelegate
- `VerifierVerdict` struct using existing `VerifyResult` type
- `ConsensusResult` with individual verdicts + agreement ratio
- Pure logic, no IO dependencies

**Tests:**
- Majority: 2/3 approved → approved, 1/3 → rejected
- Weighted: confidence-based calculation
- Unanimous: all must agree
- LeaderDelegate: leader decides
