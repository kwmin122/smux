# smux Dogfood Observations

> 2026-03-21: smux를 만들면서 동시에 smux가 필요한 상황을 경험 중.
> Claude(계획/실행) ↔ Codex(검증) 핑퐁을 수동으로 하면서 느끼는 pain point와 설계 시사점.

## 현재 워크플로우 (수동)

```
1. Claude가 태스크 완료 (코드 작성/수정)
2. 경욱님이 결과를 복사
3. Codex 터미널로 이동
4. 붙여넣기 + "이거 검증해줘" 지시 추가
5. Codex가 리뷰 결과 출력
6. 경욱님이 리뷰 결과 복사
7. Claude 터미널로 이동
8. 붙여넣기 + "이 피드백 반영해" 지시
9. 반복...
```

**Pain Points:**
- 매번 2~8 단계가 30초~1분 소요
- 복사할 때 뭘 포함하고 뭘 빼야 할지 판단 필요 (전체? 핵심만?)
- 터미널 전환이 컨텍스트 스위칭 비용 발생
- 태스크 9개 × 라운드 N회 = 수십 번 반복 예정
- **가장 큰 문제: "이제 넘겨야 할 타이밍"을 사람이 판단해야 함**

## 핵심 설계 질문: 핸드오프 트리거

### "smux는 언제 한쪽 출력을 다른쪽에 넘기는가?"

#### 관찰된 패턴 (지금 이 세션에서)

| 상황 | 트리거 | 사람이 한 일 |
|------|--------|-------------|
| Claude가 코드 커밋 | 커밋 완료 메시지 | "이거 리뷰해줘" 복사→Codex |
| Claude가 스펙 수정 | 파일 저장 완료 | "스펙 리뷰해줘" 복사→Codex |
| Codex가 리뷰 완료 | Findings 목록 출력 | 피드백 전체 복사→Claude |
| Codex가 "execution-ready" 판정 | 판정 문장 | "다음 태스크 ㄱ" |

**관찰: 트리거는 항상 "한쪽이 자기 역할을 완료한 순간"**

---

## 핸드오프 트리거 후보 4가지

### A. Always-Forward Mode (자동 전달)

```
[pingpong mode ON]
Claude 응답 완료 → smux가 자동으로 전체를 Verifier에게 전달
Verifier 응답 완료 → smux가 자동으로 전체를 Planner에게 전달
```

- 장점: 설정 한번이면 끝, 완전 자동
- 단점: 모든 응답이 넘어감 (중간 질문, 확인 등도). 노이즈 과다.
- 적합: 단순한 계획→검증 루프

### B. Signal-Based (에이전트가 신호)

```
Claude: "검증 요청합니다" (또는 JSON verdict 블록 출력)
→ smux가 이 신호를 감지 → Verifier에게 전달
```

- 장점: 에이전트가 "준비됐다"고 명시적으로 알림. 정확함.
- 단점: 시스템 프롬프트에서 신호 포맷을 강제해야 함. 에이전트가 안 보낼 수도.
- 적합: 구조화된 핑퐁 (smux의 주 유스케이스)

### C. Event-Based (특정 이벤트 감지)

```
Claude가 git commit 실행 → smux가 감지 → "커밋 완료, 검증 넘기기"
Claude가 파일 N개 수정 → smux가 감지 → "변경 완료, 검증 넘기기"
```

- 장점: 에이전트 수정 없이 동작. 실제 행동 기반.
- 단점: 커밋 없는 작업 (설계, 분석)에선 트리거 안 됨.
- 적합: 코드 수정 위주 Phase 2

### D. Hybrid (추천)

```
Phase별로 다른 트리거:
- Phase 1 (계획): Signal-based (에이전트가 "계획 완료" 출력)
- Phase 2 (실행): Event-based (커밋 감지) + Signal-based 병행
- 수동 오버라이드: [Enter] 키로 사용자가 강제 핸드오프
```

- 장점: 상황에 맞는 트리거. 수동 탈출구 있음.
- 단점: 구현 복잡도 높음.
- 적합: v0.3+ (v0.1은 Signal-based만)

## v0.1 핸드오프 설계 제안

**v0.1은 Signal-based + Manual override:**

1. Planner의 시스템 프롬프트에 삽입:
   ```
   작업이 완료되면 반드시 마지막에 다음 블록을 출력하세요:
   ```json
   {"handoff": "verify", "summary": "무엇을 했는지 한 줄"}
   ```
   ```

2. Verifier의 시스템 프롬프트에 삽입:
   ```
   검증이 완료되면 반드시 마지막에 다음 블록을 출력하세요:
   ```json
   {"verdict": "APPROVED"|"REJECTED", ...}
   ```
   ```

3. smux가 `{"handoff":` 또는 `{"verdict":` 패턴을 감지하면 자동 전달.

4. 감지 실패 시: 사용자가 [Enter]로 수동 트리거.

5. [p] 키로 일시정지 → 수동으로 컨텍스트 편집 후 전달 가능.

## 경험에서 나온 추가 인사이트

1. **복사할 때 "전체 vs 핵심"을 고르는 것도 자동화해야 함**
   - 지금 경욱님이 매번 판단: "이 리뷰 결과 전체를 보내야 하나, Findings만?"
   - smux의 context passer가 이걸 자동으로 해야 함 (이미 스펙에 4000 토큰 제한 있음)

2. **"모드"로 켜는 게 맞다**
   - `smux start --mode pingpong` 또는 세션 중 `/pingpong on`
   - 모드 OFF면 일반 터미널처럼 동작, ON이면 자동 핸드오프

3. **리뷰어의 리뷰 깊이도 조절 가능해야 함**
   - 태스크 1 리뷰: 스펙과의 정합성 깊이 검증 (무거움)
   - 태스크 2 리뷰: "빌드 되나? 테스트 통과하나?" (가벼움)
   - `--review-depth quick|standard|deep`
