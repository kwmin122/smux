### Multi-Verifier End-to-End Path

Verified producer-consumer closure across the full stack:

```
CLI --verifiers claude,codex,gemini --consensus majority
  → IPC StartSession { verifiers: [...], consensus: "majority" }
  → Daemon: merge primary + extras (dedup), create N adapters
  → Orchestrator::new_multi(): run all verifiers, consensus::resolve()
  → OrchestratorEvent::CrossVerifyResult emitted
  → Daemon: convert to DaemonMessage::CrossVerifyResult
  → Tauri: stream_daemon_events → DaemonEvent::CrossVerifyResult
  → React: setCrossVerify() → MissionControl renders per-verifier rows
  → CLI: formatted output "=== Cross-Verify (round N) ==="
```

## Round 4 Review Results

### Verification-Gate v2 (CHANGES_REQUIRED)

**Claim-to-proof: 5/5 verified** — multi-verifier e2e, primary merge, NeedsInfo re-ask, git status, CLI format.

Mediums (spec deviations, not correctness bugs):
- VG-001: Sequential verifier execution (spec says parallel `tokio::join!`)
- VG-002: Config file missing `consensus`/`verifiers` fields
- VG-003: `VerifierVerdict` field names differ from spec, `duration_ms` omitted
- VG-004: BrowserPanel uses iframe instead of Tauri WebView

Lows:
- VG-005: Single HealthMonitor shared across all verifiers
- VG-006: `_session_id` discarded in Tauri streaming
- VG-007: Planner feedback only uses first verifier output

### Security Audit

Top 3 remediation priorities:
1. **C-3**: Unix socket permissions → `0o600` + peer credential validation
2. **C-2**: Set restrictive CSP in `tauri.conf.json`
3. **H-3**: Daemon should use `create_adapter_with_safety()` to apply SafetyConfig

Other findings: 8-char session ID brute-forceable (H-1), iframe `allow-same-origin` (H-4), PID race (M-1), 16MiB message limit (M-2), terminal escape injection (M-3), no max_rounds server-side cap (M-4).
