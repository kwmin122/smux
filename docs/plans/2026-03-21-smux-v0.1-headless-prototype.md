# smux v0.1 Headless Prototype Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build a headless `smux` prototype that can run a 2-agent ping-pong loop for `claude` and `codex`, persist round snapshots, and rewind a worktree-backed session without a GUI.

**Architecture:** Start with a Rust workspace containing a testable `smux-core` crate and a thin `smux-cli` binary. For v0.1, both providers use headless structured-output modes: Claude via `-p --output-format stream-json`, Codex via `exec --json`. De-risk the build in this order: validate external CLI assumptions, implement pure core logic with fake adapters, capture and parse real headless event fixtures, add git/worktree rewind, then wire the real CLI commands.

**Tech Stack:** Rust workspace, `tokio`, `serde`, `serde_json`, `clap`, `tracing`, `tempfile`, `assert_cmd`, `insta`

## Notes for Execution

- Start with fake adapters and temp git repos. Do not make the real CLI integrations your first proof of correctness.
- Keep provider command validation from Task 1 as the single source of truth for flags and launch arguments.
- Treat recorded Claude/Codex event fixtures as the contract tests for v0.1. If the live schema differs, update fixtures first and then the parser.
- Do not add the Tauri app, browser integration, or daemon in v0.1.
- Do not add a PTY adapter in v0.1. PTY is a v0.2 fallback for providers without a usable headless mode.
- If the real Claude or Codex headless output does not match the validated notes, stop and update `docs/plans/2026-03-21-smux-v0.1-capability-notes.md` before changing the core abstractions.
